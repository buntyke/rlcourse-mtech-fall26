Part A — Why Do We Need Model-Free Methods?

Recap: The Problem with Dynamic Programming
  • In Unit II, DP found optimal policies using Bellman equations
  • DP needs the full transition model \(p(s',r \mid s,a)\) — i.e. exact probabilities for every outcome
  • In most real problems, this model is unknown or too large to store
  • Today: learn value functions from experience, without knowing the model
Monte Carlo: The Core Idea
\[ V(s) \approx \text{average of returns } G_t \text{ observed from } s \]
  • Run many episodes; record the total reward (return) from each visit to state \(s\)
  • Average those returns → estimate of \(v_\pi(s)\)
  • No model needed — just experience
  • Only works for episodic tasks (must reach a terminal state)
Episode Structure
An episode: \(S_0, A_0, R_1, S_1, A_1, R_2, \ldots, S_T\) (terminal). Return from step \(t\): \(G_t = R_{t+1} + \gamma R_{t+2} + \cdots + \gamma^{T-t-1}R_T\). Computed backwards: \(G \leftarrow \gamma G + R_{t+1}\).
Prediction vs Control
  • Prediction: given a fixed policy \(\pi\), estimate how good each state is — find \(v_\pi(s)\)
  • Control: find the best policy \(\pi^*\) that gives the most reward
  • Today we study prediction — how to evaluate a policy from samples
  • Control (using MC to improve a policy) is covered in the next lecture

GPI: Evaluate ↔ Improve

Visits and Returns
A visit to state \(s\): any timestep \(t\) where \(S_t = s\). A first visit: the first time \(s\) appears in an episode. Returns are computed by working backwards from the terminal step.

Part B — Monte Carlo Prediction Algorithm

First-Visit MC Prediction — Algorithm
Input: policy π Initialize: V(s) ← 0, Returns(s) ← [ ] for all states s Loop (for each episode): Play episode following π → S₀,A₀,R₁, S₁,A₁,R₂, …, S_T G ← 0 For t = T-1, T-2, …, 0: G ← γG + R_{t+1} If S_t appears for the FIRST TIME in this episode: Append G to Returns(S_t) V(S_t) ← mean( Returns(S_t) )
  • Only the return from the first visit to each state is used per episode
  • More episodes → more accurate estimate of \(v_\pi(s)\)
First-Visit vs Every-Visit MC
  • First-visit MC: use only the return from the first visit to \(s\) in each episode
  • Every-visit MC: use returns from all visits to \(s\) — remove the "first time" check
  • Both methods give correct estimates as episodes → ∞
  • Every-visit is simpler to code and works better with function approximation
How MC Estimates Converge
By the Law of Large Numbers: average of many i.i.d. samples converges to the true mean. Each first-visit return is an independent sample of \(v_\pi(s)\). Error decreases as \(1/\sqrt{n}\) where \(n\) = number of visits.
Efficient Update: Incremental Average
Instead of storing all returns and re-computing the mean:
\[ V(s) \leftarrow V(s) + \frac{1}{n}\bigl[G - V(s)\bigr] \]
  • Same result as computing the mean from scratch — but uses \(O(1)\) memory
  • For non-stationary tasks: replace \(\tfrac{1}{n}\) with a fixed step-size \(\alpha\): \(V(s) \leftarrow V(s) + \alpha[G - V(s)]\)
  • This same idea is used in TD learning (next lecture)
🖊️ Board: Show how the incremental mean formula is derived from the running average identity.
Worked Example: MC Prediction on a Small MDP
Chain MDP: A → B → C → Terminal, rewards 0, 0, +1, \(\gamma = 1\). After many episodes, \(V(A) = V(B) = V(C) = 1\). Returns are calculated backwards and averaged episode by episode.
Three Advantages of MC over Dynamic Programming
1. No model needed — learn directly from experience. 2. Works with a simulator that generates sample transitions (not full probability tables). 3. Can estimate value of one state without evaluating all states.

Part C — Blackjack: A Monte Carlo Case Study

Blackjack as a Reinforcement Learning Problem
  • Goal: get cards summing to ≤ 21, closer to 21 than the dealer
  • Actions: Hit (draw a card) or Stick (stop)
  • Reward: +1 win, −1 lose, 0 draw — only at end of game
  • Simulating games is easy; computing exact probabilities is hard → perfect for MC
Blackjack State Space — 200 States
State = (player's sum: 12–21) × (dealer's visible card: Ace–10) × (usable ace: Yes/No). Total: \(10 \times 10 \times 2 = 200\) states. A usable ace is one that can be counted as 11 without going over 21.
How a Blackjack Episode Works
Evaluation policy \(\pi\): stick if player sum ≥ 20, else hit. Dealer policy: stick if sum ≥ 17, else hit. No need to compute \(p(s',r|s,a)\) — just simulate and record the outcome.
Blackjack Value Function — 10,000 Episodes
Figure 5.1 (Sutton & Barto) — left column. After 10,000 simulated games the surface is noisy but the trend is clear: higher player sum → higher value. Usable-ace states have more uncertainty (visited less often).
Blackjack Value Function — 500,000 Episodes
Figure 5.1 (Sutton & Barto) — right column. After 500,000 games the surface is smooth — the Law of Large Numbers has done its job. The no-usable-ace surface is more regular because those states are visited more often.
Why MC is Better than DP for Blackjack
Even though the rules of blackjack are fully known, computing \(p(s',r|s,a)\) requires listing all possible card combinations — tedious and error-prone. MC avoids this: just play the game and record the result.

Part D — Properties of Monte Carlo Methods

Backup Diagrams: DP vs MC
DP backup: one step, all possible next states (needs the model \(p\)). MC backup: one sampled path, all the way to the terminal state (no model needed). The depth and breadth of the diagram show exactly what information each method uses.
MC Does Not Bootstrap

DP update (bootstraps)

\[ V(S_t) \leftarrow V(S_t) + \alpha\bigl[R_{t+1} + \gamma\,\textcolor{red}{V(S_{t+1})} - V(S_t)\bigr] \] Uses the estimated value of the next state

MC update (no bootstrap)

\[ V(S_t) \leftarrow V(S_t) + \alpha\bigl[\textcolor{green}{G_t} - V(S_t)\bigr] \] Uses the actual observed return
  • Bootstrapping = using your own (possibly wrong) estimates to update other estimates
  • MC never bootstraps — each state's estimate is based purely on real rewards
State Estimates are Independent
DP: all state values are coupled through Bellman equations — changing one affects others. MC: each state's estimate comes from its own independent episodes — no coupling. Result: you can estimate \(V(s)\) for any single state without evaluating all states.
The Soap Bubble Analogy
Height of a soap surface at any interior point = average of neighbouring heights. DP: solve a grid of equations (like iterative policy evaluation). MC: do random walks from the point of interest to the boundary; average boundary heights. Both give the same answer — but MC is faster when you only need one point.
MC vs DP — Summary Table
Key differences: MC needs no model, does not bootstrap, updates per episode, and can focus on a subset of states. DP needs the full model but can be more sample-efficient when all state values are required.

Part E — Estimating Action Values & Exploration

Why We Need Action Values \(Q(s,a)\)

With a model

\(\pi'(s) = \arg\max_a \sum_{s',r} p(s',r|s,a)[r + \gamma V(s')]\)
Needs \(p\) — not available

Without a model

\(\pi'(s) = \arg\max_a\, Q(s,a)\)
Only needs \(Q\) — model-free ✓
  • State values alone are not enough when the model is unknown
  • Estimating \(Q(s,a)\) directly lets us improve the policy without any model
How to Estimate \(Q(s,a)\) with MC
Change Returns(s) → Returns(s, a) Change V(S_t) → Q(S_t, A_t) Check: is (S_t, A_t) the first visit to this pair in this episode? Update: Q(S_t, A_t) ← mean( Returns(S_t, A_t) )
  • A visit to \((s,a)\): any timestep where \(S_t=s\) and \(A_t=a\)
  • Return \(G_t\) is associated with the pair \((s,a)\), not just the state \(s\)
  • Everything else is the same as MC prediction for \(V(s)\)
The Exploration Problem
If the policy always picks the same action in a state, other actions are never tried. We get no data for \(Q(s, \text{other actions})\) and cannot compare them. This is the same explore-exploit tension seen in the bandit problem (Unit I).
Solution 1: Exploring Starts
  • Start each episode from a randomly chosen state-action pair \((s,a)\)
  • Every pair must have a non-zero chance of being chosen as the start
  • Over many episodes, all pairs will be visited and estimated
  • Limitation: only works in simulation — in real settings we cannot control the starting state
Solution 2: Soft (Exploratory) Policies
\(\varepsilon\)-greedy policy: pick the best action most of the time,
but with small probability \(\varepsilon\) pick a random action instead
  • Every action always has some probability of being chosen → all \((s,a)\) pairs eventually visited
  • No special starting conditions needed — works in real environments too
  • Tradeoff: finds the best exploratory policy, not the true optimal
  • Next lecture: on-policy and off-policy MC control using \(\varepsilon\)-greedy
Backup Diagram for \(Q(s,a)\)
The \(Q\) backup diagram starts at a state-action pair (filled square), not just a state. Below it, the same sampled trajectory as the \(V\) backup goes all the way to the terminal state. Return \(G\) accumulates from bottom to top along this single trajectory.
General Policy Iteration with Monte Carlo
\[\pi_0 \xrightarrow{E} q_{\pi_0} \xrightarrow{I} \pi_1 \xrightarrow{E} q_{\pi_1} \xrightarrow{I} \cdots \to \pi_*\]
  • E step: run episodes, average returns → estimate \(Q\) for current policy
  • I step: update policy to be greedy w.r.t. \(Q\): \(\pi(s) \leftarrow \arg\max_a Q(s,a)\)
  • Alternate E and I after each episode — same GPI loop as in DP
  • Converges to optimal policy when all state-action pairs are explored
Summary & What Comes Next
  • MC estimates value by averaging returns from sample episodes — no model needed
  • First-visit and every-visit MC both converge; no bootstrapping
  • Estimate \(Q(s,a)\) for model-free control; need exploration to cover all pairs
  • Next lecture: MC Control — Monte Carlo ES, on-policy \(\varepsilon\)-soft, off-policy (importance sampling)
